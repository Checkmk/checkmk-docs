[#intro]
== Introduction

ifdef::onprem[]
{cce-only}
endif::[]
There are cases where a {CMK} site cannot reach the host, for example when firewall configuration blocks incoming traffic from a network.
With
ifdef::onprem[]
{UE}, 
endif::[]
ifdef::saas[]
{CE}, 
endif::[]
the {CMK} Relay can be deployed to collect monitoring data and send it to the monitoring site.


[#relaydiffs]
=== Differences from distributed monitoring

There are notable differences compared to xref:glossar#distributed_monitoring[distributed monitoring]:

* _The communication direction_: The Relay always xref:ports#relay[sends data] to the monitoring site.
* _The place of data processing_: The Relay neither processes, nor stores data, it just collects monitoring data and passes it on.

ifdef::onprem[]
Using the Relay in _combination_ with distributed monitoring is currently limited to the central site.
This also means that there is typically no practical use for it in environments using {ME} as intended. 
endif::[]


[#scope]
=== Scope of functionality

The current state of the Relay encompasses functionality as follows:

* A growing selection of xref:glossar#special_agent[special agents]
* Monitoring via xref:glossar#snmp[SNMP]

Work on extending functionality towards running active checks is currently under way.
The prioritizing of special agent additions is dependent on user benefit and the effort needed.
A list of the special agents that are included in the Relay is available at link:https://github.com/Checkmk/checkmk/blob/{current-major}/relay_supported_plugins.bzl[GitHub^].
ifdef::onprem[]
Special agents deployed via xref:glossar#mkp[MKP] to the {CMK} site are _not_ synced to the Relay.
endif::[]


[#setup]
== Setup


[#prerequisites]
=== Technical requirements


[#ossupport]
==== Supported operating systems
The {CMK} Relay is provided as a link:https://hub.docker.com/r/checkmk/check-mk-relay[container^] for the _x86-64_ platform.
The following operating systems are currently supported:

* Ubuntu Linux 24.04
* Red Hat Enterprise Linux 8.10+, 9.7+, and 10
* Windows Server 2022 and 2025

On Windows, the {CMK} Relay runs on the _Windows Subsystem for Linux_ version 2 (WSL2).
To prepare the necessary environment for this, an Ubuntu distribution is set up in WSL during the xref:setup[installation] process.

If you run Windows in a virtual machine, this system must enable nested virtualization in order for the Relay to work as expected.


==== Podman and connection to Docker Hub

For the installation we provide a script that requires the link:https://podman.io/[Podman^] package provided by your distribution to be installed.
The required container image is downloaded directly from Docker Hub during installation and with every update.
Make sure your target system can connect to Docker Hub.

The xref:setup[installation] wizard on Linux systems includes a step for explicitly installing Podman.
When installing on Windows, the MSI installer we provide also handles the installation of Podman within the WSL environment.

We chose Podman because its tight integration with Systemd facilitates automatic updates of the Relay container.
Deploying on container runtimes other than Podman, running on distributions other than those listed, updating via methods other than direct access to `hub.docker.com`, using Podman packages other than those provided by the distribution, or using multiple Relays on a single Linux host may work for experienced users--however, {CMK} does not provide support for such environments.

You can run the installation script on distributions other than those listed above.
If compatible versions of Podman and Systemd are found, the script will prompt you to either abort or proceed with an unsupported installation.

==== Windows: Connection to GitHub

An additional requirement under Windows is that a connection to GitHub must be possible at installation time.
This connection is needed in order to download and install the latest version of WSL2.

[#hwrequirements]
==== Hardware requirements

In the standard configuration (13 fetchers), you should expect RAM requirements of about one gigabyte and CPU core utilization of less than 20{nbsp}% on current hardware.
With these resources, we were able to monitor more than 50 SNMP devices without any problems.
If the number of devices to be monitored is significantly higher, the number of fetchers may need to be increased.
As the number of devices increases, resource requirements scale almost linearly with the number of devices being monitored.

For each Relay registered on a site, the number of fetchers can be customized via [.guihint]#Setup > Hosts > Relays# by clicking the edit icon (icon:icon_edit[alt="Edit icon"]).


[#setup_relay]
=== Configuring and installing a Relay

To set up a Relay, navigate to [.guihint]#Setup > Hosts > Relays#.
There, click [.guihint]#Add Relay configuration# to launch the setup wizard for configuring a new Relay.
First, choose between [.guihint]#Linux# and [.guihint]#Windows# to get the appropriate instructions for your target system.

.In the first step, the download link for the installation script is provided
image::relay_wizard.png[alt="In the Quick Setup for a new Relay, this step displays the curl command to download the installation script."]

Follow the wizard's instructions.

[.tabs]
====
[#wizard_linux]#Linux#::
+
--
In the wizard for Linux systems, installing Podman is a separate step that you perform individually on the target system.

Afterward, you can run the provided installation script as described in the wizard.
When you run the script, the container with the Relay is downloaded and set up, the update service is installed, and the Relay is connected to the {CMK} site--all in a single action.

On Linux, no separate log file is created for the installation.
If needed, you can redirect the output of the installation script to a file.
--
[#wizard_windows]#Windows#::
+
--
In the Windows wizard, you first prepare the WSL environment on the target system.
This step may take a few minutes and requires a reboot of the system.

You can then download and run the provided MSI installer as described in the wizard.
When the script runs, Podman is installed first, followed by the download and setup of the container containing the Relay, the installation of the update service, and the connection of the Relay to the {CMK} site--all in a single operation.

If necessary, you can check the installation script's log file to verify that the installation proceeded as expected.
On Windows, you can find this log file in the `C:\ProgramData\checkmk\relay\logs` folder.
--
====

[TIP]
====
To register the Relay with your {CMK} site, the wizard will prompt you to complete the installation using a _one-time token_.
This token is valid for one hour after it is generated.
ifdef::onprem[]
If one hour is not sufficient, you can also complete the registration using the username and password of a user who has sufficient permissions to register the Relay.
endif::[]
ifdef::saas[]
If one hour is not sufficient, you can also complete the registration using the username and password of an automation user who has sufficient permissions to register the Relay.
endif::[]
====

In the {CMK} site to which you just added the Relay, you must now xref:glossar#activate_changes[activate pending changes] to complete its registration.


[#reinstall]
==== Notes on reinstallation
If you install and register the Relay multiple times on the same system, your {CMK} site will treat each installation as a new Relay.
Hosts that were already being monitored by the previous Relay must then be explicitly assigned to the newly registered Relay--even if the Relay is running on the same machine as before.


[#updates]
=== Updates

A {CMK} Relay deployed using the provided installation script automatically keeps itself up to date.
As soon as it detects that the version of the associated {CMK} site has changed, it downloads the appropriate container image from Docker Hub and restarts without requiring manual intervention.
Depending on your network speed, this process can take up to 10 minutes.

If you encounter issues with incomplete updates, please refer to our xref:update_manager[troubleshooting guidelines].


[#monitoring_relay]
=== Monitoring the Relay

If you want to monitor the container in which you are running the Relay, you can add it to your monitoring using the xref:monitoring_podman#[`mk_podman` agent plug-in].


[#monitoring]
== Monitoring with the Relay

Once you have set up the Relay, you can now start using it to monitor hosts via a special agent or SNMP.
When adding a host, enable [.guihint]#Monitored on Relay# in the [.guihint]#Basic settings# section and select the Relay that can contact the host you are currently adding.
Note that hosts assigned to a Relay cannot be monitored with the {CMK} agent.

The Relay will not begin monitoring the assigned hosts until special agents or SNMP connections have been configured for those hosts.

Each Relay periodically contacts its assigned {CMK} site for configuration updates.
After receiving an updated configuration, it immediately executes the special agents and other checks.
A xref:glossar#service_discovery[service discovery] for special agents should be completed quickly, but it may take longer with slow SNMP devices.
In general, the delay caused by the Relay is marginal and, in most cases, is in the range of a few seconds.

[#version]
=== Verifying the version of the Relay

The version of the Relay container image must exactly match the version of the {CMK} site to which it is assigned.
You can verify the Relay version using the `podman images` command.
Run this command as root or with `sudo` prefixed to ensure that the output includes all relevant images:

[{shell}]
----
{c-root} podman images

[#restart]
=== Restarting the container

As mentioned earlier, Podman is tightly integrated with Systemd.
Therefore, restarting containers is simply a matter of restarting Systemd units.

[{shell}]
----
{c-root} systemctl restart checkmk_relay
{c-root} systemctl status checkmk_relay
----


[#update_manager]
=== Status of the update service

When deploying using the included installation script, the {CMK} Relay sets up a service that automatically updates the container image to the same version as the associated site and then restarts the container.

To check the status of the update service for the {CMK} Relay, run the following command:

[{shell}]
----
{c-user} systemctl status checkmk_relay-update-manager
○ checkmk_relay-update-manager.service - Checkmk Relay Update Manager
     Loaded: loaded (/etc/systemd/system/checkmk_relay-update-manager.service; static)
     Active: inactive (dead)
TriggeredBy: ● checkmk_relay-update-manager.path
----

The service has been loaded here, but no update has been performed yet.
Since the update itself is only triggered when a file appears, the service displays the (misleading) status `inactive (dead)`.

If you notice that the Relay is in an error state because an update could not be performed, you can resolve this issue manually.
First, reset the error status of the update service:

[{shell}]
----
{c-root} systemctl reset-failed checkmk_relay-update-manager.service checkmk_relay-update-manager.path
----

After that, you can restart the update service:

[{shell}]
----
{c-root} systemctl restart checkmk_relay-update-manager.path
----

Finally, check to make sure the status of the update service is okay after the restart.

[{shell}]
----
{c-user} systemctl status checkmk_relay-update-manager.path checkmk_relay-update-manager.service --no-pager
----

If you manually restart the service and check its status immediately afterward, you may see output similar to the following:

[{shell}]
----

[#uninstall_windows]#Windows#::

In Windows, you can uninstall the Relay via [.guihint]#Settings > Apps > Installed Apps#.
Select the {CMK} Relay from the list of installed programs and click [.guihint]#Uninstall#.

As long as the downloaded MSI installer is present on the system, you can use it to perform the uninstallation via PowerShell instead.
This step does not take place within the WSL environment, but directly in Windows.

[{powershell}]
----
PS C:\Users\hhirsch> msiexec /x checkmk-relay.msi /passive
----

--
====

Don't forget to delete the associated Relay in your {CMK} site after you have removed the Relay.
