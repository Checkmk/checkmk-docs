[#intro]
== Introduction

ifdef::onprem[]
{cce-only}
endif::[]
There are cases where a {CMK} site cannot reach the host, for example when firewall configuration blocks incoming traffic from a network.
With
ifdef::onprem[]
{UE}, 
endif::[]
ifdef::saas[]
{CE}, 
endif::[]
the {CMK} Relay can be deployed to collect monitoring data and send it to the monitoring site.

include::include_technical_preview.asciidoc[]


[#relaydiffs]
=== Differences from distributed monitoring

There are notable differences compared to xref:glossar#distributed_monitoring[distributed monitoring]:

* _The communication direction_: The relay always xref:ports#relay[sends data] to the monitoring site.
* _The place of data processing_: The relay neither processes, nor stores data, it just collects monitoring data and passes it on.

ifdef::onprem[]
Using the relay in _combination_ with distributed monitoring is currently limited to the central site.
This also means that there is typically no practical use for it in environments using {ME} as intended. 
endif::[]


[#scope]
=== Scope of functionality

The current state of the relay encompasses functionality as follows:

* A growing selection of xref:glossar#special_agent[special agents]
* Monitoring via xref:glossar#snmp[SNMP]

Work on extending functionality towards running active checks is currently under way.
The prioritizing of special agent additions is dependent on user benefit and the effort needed.
A list of the special agents that are included in the relay is available at link:https://github.com/Checkmk/checkmk/blob/{current-major}/relay_supported_plugins.bzl[GitHub^].
ifdef::onprem[]
Special agents deployed via xref:glossar#mkp[MKP] to the {CMK} site are _not_ synced to the relay.
endif::[]

[#setup]
== Einrichtung

[#prerequisites]
=== Technische Voraussetzungen

[#ossupport]
==== Unterstützte Betriebssysteme
Das {CMK} Relay wird als link:https://hub.docker.com/r/checkmk/check-mk-relay[Container^] für die Plattform _x86-64_ bereitgestellt.
Derzeit werden die folgenden Betriebssysteme unterstützt:

* Ubuntu Linux 24.04
* Red Hat Enterprise Linux 8.10+, 9.7+ und 10
* Windows Server 2022 und 2025

Beim Betrieb unter Windows läuft das {CMK} Relay auf dem _Windows Subsystem for Linux_ in der Version 2 (WSL2).
Dazu wird im Zuge der xref:setup[Installation] eine Ubuntu-Distribution in WSL aufgesetzt.


==== Podman und Verbindung zu Docker Hub

Für die Installation stellen wir ein Skript bereit, das voraussetzt, dass das von der Distribution angebotene link:https://podman.io/[Podman^]-Paket vorhanden ist.
Das benötigte Image für den Container wird bei der Installation und bei jedem Update direkt von Docker Hub heruntergeladen.
Stellen Sie also sicher, dass Ihr Zielsystem eine Verbindung zu Docker Hub aufbauen kann.

Der Assistent für die xref:setup[Installation] auf Linux-Systemen enthält einen Schritt für die explizite Installation von Podman.
Bei der Installation unter Windows kümmert sich der von uns bereitgestellte MSI-Installer auch um die Installation von Podman innerhalb der WSL-Umgebung.

Wir haben uns für Podman entschieden, weil dessen enge Integration mit Systemd automatische Updates des Relay-Containers erleichtert. 
Das Ausbringen auf anderen Container-Runtimes als Podman, der Betrieb auf anderen als den genannten Distributionen, die Aktualisierung auf anderen Wegen als dem direkten Zugriff auf `hub.docker.com`, die Nutzung mit anderen Podman-Paketen als denen der Distribution oder die Verwendung von mehreren Relays auf einem Linux-Host mag für erfahrene Nutzer funktionieren – allerdings bietet {CMK} keinen Support für derartige Umgebungen an.

Sie können das Installationsskript auf anderen als den oben genannten Distributionen ausführen.
Werden kompatible Versionen von Podman und Systemd vorgefunden, wird Sie das Skript vor die Wahl stellen, entweder abzubrechen oder eine nicht unterstützte Installation durchzuführen.



[#hwrequirements]
==== Hardware-Anforderungen

In der Standardkonfiguration (13 Fetcher) sollte ein Arbeitsspeicherbedarf von etwa einem Gigabyte und die Auslastung eines Prozessorkerns aktueller Hardware mit weniger als 20{nbsp}% eingeplant werden.
Mit diesem Ressourcenumfang konnten wir problemlos mehr als 50 SNMP-Geräte überwachen.
Bei einer deutlich höheren Zahl zu überwachender Geräte muss gegebenenfalls die Zahl der Fetcher erhöht werden.
Nach oben skaliert der Ressourcenbedarf nahezu linear mit der Zahl zu überwachender Geräte.

Für jedes Relay, das bei einer Instanz registriert ist, kann die Anzahl der Fetcher über [.guihint]#Setup > Hosts > Relays# mit einem Klick auf das Symbol zum Bearbeiten (icon:icon_edit[alt="Symbol für das Bearbeiten."]) angepasst werden.

[#setup_relay]
=== Relay konfigurieren und installieren

Zum Einrichten eines Relays navigieren Sie zu [.guihint]#Setup > Hosts > Relays#.
Klicken Sie dort auf [.guihint]#Add Relay configuration#, um den Einrichtungsassistenten zur Konfiguration eines neuen Relays zu starten.
Wählen Sie zuerst zwischen [.guihint]#Linux# und [.guihint]#Windows#, um die passenden Anweisungen für Ihr Zielsystem zu erhalten.

.Im ersten Schritt wird der Downloadlink des Installationsskripts bereitgestellt
image::relay_wizard.png[alt="Im Quick Setup für ein neues Relay zeigt der Schritt den Curl-Befehl zum Herunterladen des Installationsskripts."]

Folgen Sie den Anweisungen des Assistenten.

[.tabs]
====
[#wizard_linux]#Linux#::
+
--
Im Assistenten für Linux-Systeme ist die Installation von Podman ein separater Schritt, den Sie einzeln auf dem Zielsystem ausführen.

Danach können Sie das bereitgestellte Installationsskript wie im Assistenten beschrieben ausführen.
Beim Ausführen des Skripts wird der Container mit dem Relay heruntergeladen und eingerichtet, der Dienst für die Aktualisierung installiert und das Relay mit der {CMK}-Instanz verbunden -- in einem einzigen Schritt.

Unter Linux wird keine eigene Logdatei für die Installation angelegt.
Bei Bedarf können Sie die Ausgabe des Installationsskripts in eine Datei umleiten.
--
[#wizard_windows]#Windows#::
+
--
Im Assistenten für Windows bereiten Sie zunächst die WSL-Umgebung auf dem Zielsystem vor.
Dieser Schritt kann einige Minuten dauern.

Danach können Sie den bereitgestellten MSI-Installer wie im Assistenten beschrieben herunterladen und ausführen.
Beim Ausführen des Skripts wird zunächst Podman installiert und dann der Container mit dem Relay heruntergeladen und eingerichtet, der Dienst für die Aktualisierung installiert und das Relay mit der {CMK}-Instanz verbunden -- in einem einzigen Schritt.

Bei Bedarf können Sie in der Logdatei des Installations-Skriptes nachsehen, ob die Installation wie erwartet verlaufen ist.
Diese Logdatei finden Sie unter Windows im Ordner `C:\ProgramData\checkmk\relay\logs`.
--
====

[TIP]
====
Um das Relay bei Ihrer {CMK}-Instanz zu registrieren, weist der Assistent Sie an, die Installation mit einem Einmal-Token (_one-time token_) auszuführen.
Dieses ist nach Erzeugung eine Stunde lang gültig.
ifdef::onprem[]
Sollte diese Stunde nicht ausreichend sein, können Sie die Registrierung auch mit Benutzername und Passwort eines Benutzers vornehmen, der ausreichende Berechtigungen für die Registrierung des Relays hat.
endif::[]
ifdef::saas[]
Sollte diese Stunde nicht ausreichend sein, können Sie die Registrierung auch mit Benutzername und Passwort eines Automationsbenutzers vornehmen, der ausreichende Berechtigungen für die Registrierung des Relays hat.
endif::[]
====

In der {CMK}-Instanz, zu der Sie eben das Relay hinzugefügt haben, müssen Sie jetzt noch die xref:glossar#activate_changes[Änderungen aktivieren], um dessen Registrierung abzuschließen.

[#reinstall]
==== Hinweise zur Neuinstallation
Wenn Sie die Installation und Registrierung des Relays auf dem gleichen System mehrfach ausführen, wertet Ihre {CMK}-Instanz das jedes Mal als neues Relay.
Hosts, die bereits über das vorige Relay überwacht wurden, müssen dann explizit dem neu registrierten Relay zugeordnet werden -- auch, wenn das Relay auf der gleichen Maschine läuft wie bisher.

[#updates]
=== Updates

Ein mit dem bereitgestellten Installationsskript verteiltes {CMK} Relay hält sich automatisch auf dem neuesten Stand.
Sobald es erkennt, dass sich die Version der zugehörigen {CMK}-Instanz geändert hat, lädt es das passende Container Image von Docker Hub herunter und startet ohne manuellen Eingriff neu.
Abhängig von Ihrer Netzwerkgeschwindigkeit kann dieser Vorgang bis zu 10 Minuten dauern.

Bei Problemen mit unvollendeten Updates beachten Sie bitte unsere xref:update_manager[Hinweise zum Troubleshooting].

[#monitoring_relay]
=== Das Relay überwachen

Wenn Sie den Container überwachen möchten, in dem Sie das Relay betreiben, können Sie ihn über das xref:monitoring_podman#[Agentenplugin `mk_podman`] zum Monitoring hinzufügen.

[#monitoring]
== Monitoring mit dem Relay

Nachdem Sie das Relay eingerichtet haben, können Sie nun damit loslegen, es zur Überwachung von Hosts via Spezialagent oder SNMP zu nutzen.
Beim Hinzufügen eines Hosts aktivieren Sie [.guihint]#Monitored on Relay# im Abschnitt [.guihint]#Basic settings# und wählen Sie das Relay aus, welches den Host kontaktieren kann, den Sie gerade hinzufügen.
Beachten Sie dabei, dass Hosts, die einem Relay zugeordnet sind, nicht mit dem {CMK}-Agenten überwacht werden können.

Das Relay wird mit der Überwachung der zugeordneten Hosts erst dann beginnen, wenn Spezialagenten oder SNMP-Verbindungen für diese Hosts konfiguriert sind.

Jedes Relay kontaktiert die ihm zugeordnete {CMK}-Instanz regelmäßig für Konfigurations-Updates.
Nachdem eine aktualisierte Konfiguration erhalten wurde, führt es die Spezialagenten und andere Checks sofort aus.
Eine xref:glossar#service_discovery[Service-Erkennung] für Spezialagenten sollte schnell abgeschlossen sein, bei langsamen SNMP-Geräten kann es jedoch länger dauern.
Generell ist die Verzögerung durch das Relay marginal und liegt in den meisten Fällen im Bereich weniger Sekunden.


[#troubleshooting]
== Troubleshooting

Zur Erinnerung: Diese Komponente von {CMK} ist Work-in-Progress!
Obwohl schon recht stabil, ist es möglich, dass Sie auf den einen oder anderen Bug stoßen.

Bei technischen Problemen kann es nützlich sein, das Verhalten und den Zustand des Relays und des Containers, in dem es läuft, zu untersuchen.
Unter Windows wechseln Sie dafür zunächst in die WSL-Umgebung:

[{powershell}]
----
PS C:\Users\hhirsch> wsl.exe -d Ubuntu-Checkmk-Relay
----

Auf Linux-Systemen ist kein solcher Wechsel nötig.

Alle Befehle, die im Folgenden gezeigt werden, können auf Linux-Systemen direkt ausgeführt werden und auf Windows-Systemen nach dem Wechsel in die WSL-Umgebung.

[#restart]
=== Neustart des Containers

Wie bereits erwähnt, ist Podman eng mit Systemd integriert.
Daher sind Neustarts von Containern einfach Neustarts von Systemd Units.

[{shell}]
----
{c-root} systemctl restart checkmk_relay
{c-root} systemctl status checkmk_relay
----

[#update_manager]
=== Zustand des Aktualisierungsdienstes

Bei der Verteilung über das mitgelieferte Installationsskript richtet das {CMK} Relay einen Dienst ein, der das Container Image automatisch auf die gleiche Version wie die zugehörige Instanz aktualisiert und den Container anschließend neu startet.

Um den Status des Aktualisierungsdienstes für das {CMK} Relay zu überprüfen, führen Sie folgenden Befehl aus:

[{shell}]
----
{c-user} systemctl status checkmk_relay-update-manager
○ checkmk_relay-update-manager.service - Checkmk Relay Update Manager
     Loaded: loaded (/etc/systemd/system/checkmk_relay-update-manager.service; static)
     Active: inactive (dead)
TriggeredBy: ● checkmk_relay-update-manager.path
----

Hier ist der Dienst geladen, es wurde jedoch noch keine Aktualisierung durchgeführt.
Da die Aktualisierung selbst erst getriggert wird, wenn eine Datei auftaucht, zeigt der Dienst den (irreführenden) Status `inactive (dead)`.


Falls Sie feststellen, dass das Relay in einem Fehlerzustand ist, weil ein Update nicht durchgeführt werden konnte, können Sie dieses Problem manuell beheben.
Setzen Sie zunächst den Fehlerzustand des Aktualisierungsdienstes zurück:

[{shell}]
----
{c-root} systemctl reset-failed checkmk_relay-update-manager.service checkmk_relay-update-manager.path
----

Danach können Sie den Aktualisierungsdienst neu starten:

[{shell}]
----
{c-root} systemctl restart checkmk_relay-update-manager.path
----

Prüfen Sie zum Abschluss, ob der Status des Aktualisierungsdienstes nach dem Neustart in Ordnung ist.

[{shell}]
----
{c-user} systemctl status checkmk_relay-update-manager.path checkmk_relay-update-manager.service --no-pager
----


Wenn Sie den Dienst manuell neu starten und den Status unmittelbar danach überprüfen, werden Sie möglicherweise eine Ausgabe wie die folgende zu sehen bekommen:

[{shell}]
----
{c-user} systemctl status checkmk_relay-update-manager
● checkmk_relay-update-manager.service - Checkmk Relay Update Manager

[#platforms]
=== Supported platforms

The {CMK} Relay is distributed as a link:https://hub.docker.com/r/checkmk/check-mk-relay[container^] for the `x86-64` platform.


[#ossupport]
==== Operating system support

For installation, we provide an installation script that requires the distribution-supplied link:https://podman.io/[Podman^] package to be present.
Currently, the installation script supports Ubuntu Linux 24.04 as well as Red Hat Enterprise Linux 8.10, 9.7 and 10.

We decided for Podman since its tight integration with Systemd facilitates automatic updates of the relay container.
Deployment on other container platforms than Podman, on other distributions than those mentioned, updating using other ways than directly accessing `hub.docker.com`, using other Podman packages than the distribution supplied ones or running multiple relays on a single Linux host might work for experienced users -- however, {CMK} will not provide support for such environments.

.The {CMK} Relay on Windows
[%collapsible]
====
We are currently in progress testing the relay on Windows.
To be precise: on _Windows Subsystem for Linux_ (aka WSL).
So far, we had positive results with WSL2, with Systemd enabled, using the latest stable WSL2 (2.6.3 as of the time this text last being edited) and running Ubuntu 24.04 atop.

Windows Server 2025 enables all dependencies needed by just running:

[{powershell}]
----
PS C:\Windows\system32> wsl.exe --install
PS C:\Windows\system32> wsl.exe --update
PS C:\Windows\system32> wsl.exe --install -d Ubuntu
----

For older Windows versions, you have to manually ensure that WSL2 is selected and Systemd is used as init system.
We will create check and installation scripts during the months after {CMK} {v25} release.
====


[#hwrequirements]
==== Hardware requirements

In the standard configuration (13 fetchers), you should expect memory requirements of approximately one gigabyte and a CPU utilization of less than 20{nbsp}% of one core on current hardware.
With these resources, we were able to monitor more than 50 SNMP devices without any issues.
If the number of devices to be monitored is significantly higher, you may need to increase the number of fetchers.
For a higher number of devices to be monitored, resource requirements scale almost linearly.

For easy download, installation and association, we provide an installation script.
To retrieve the URL of the installation script, navigate to [.guihint]#Setup > Hosts > Relays#.
There, click the [.guihint]#Add Relay configuration# button to start the assistant for configuring a new relay.

.In the first step, you are presented with the download URL of the installation script
image::relay_wizard.png[alt="Quick Setup for a new relay showing the first step with a curl command to download the installation script."]

Follow the instructions the assistant provides.
After some preparatory steps, you will be presented with the parameters to run the script.
This will download and configure the container with the relay, install the update service and associate the relay with the {CMK} site in a single step.

[#updates]
=== Updates

A {CMK} Relay that has been deployed with the provided installation script keeps itself up to date automatically.
Once it detects that the version of the associated {CMK} site has changed, it pulls the matching container image from Docker Hub and restarts without any manual intervention.
Depending on your network speed, this process can take up to 10 minutes.


[#monitoring]
== Monitoring via relay

After configuring your relay, you are now all set to use it to start monitoring hosts via special agents or SNMP.
When adding a host, activate [.guihint]#Monitored on Relay# in the [.guihint]#Basic settings# box and select the relay that can connect to the host you are adding.
Keep in mind that monitoring hosts with the {CMK} agent is currently not possible if the hosts are assigned to a relay.

The relay will only start monitoring the hosts assigned to it once the special agents or SNMP connections for those hosts are configured.

Each relay contacts its associated {CMK} site frequently for configuration updates and runs special agents and other checks immediately after pulling an updated configuration.
A xref:glossar#service_discovery[service discovery] for special agents should complete quickly, though sluggish SNMP devices may take longer.
In general, the delay caused by the relay is minimal and, in most cases, amounts to just a few seconds.

[#version]
=== Check the version of the relay

The version of the relay container image must match the exact version of the {CMK} site it is connected to.
You can check the version with the `podman images` command.
Running this command as root or prefaced with `sudo` ensures that the list includes all relevant images:

[{shell}]
----
{c-root} podman images
REPOSITORY                        TAG           IMAGE ID      CREATED       SIZE
docker.io/checkmk/check-mk-relay  {current-major}b1       423f9b67558b  5 months ago  428 MB
localhost/checkmk_relay           checkmk_sync  72c27637aa5a  5 months ago  428 MB
docker.io/checkmk/check-mk-relay  {current-major}b2       72c27637aa5a  5 months ago  428 MB
----

When deployed with the supplied installation script, the {CMK} Relay installs a service that automatically updates the container image to the same version as the associated site and restarts the container afterwards.

To check the state of the update manager for the {CMK} Relay, run:

[{shell}]
----
{c-user} systemctl status checkmk_relay-update-manager
○ checkmk_relay-update-manager.service - Checkmk Relay Update Manager
     Loaded: loaded (/etc/systemd/system/checkmk_relay-update-manager.service; static)
     Active: inactive (dead)
TriggeredBy: ● checkmk_relay-update-manager.path
----

Here, the service is loaded, but no update has been performed yet.
Since the update itself is only triggered when a file appears, the service has the (misleading) state `inaktive (dead)`.

When manually restarting the service and checking the status immediately afterwards, you might encounter an output like this:

[{shell}]
----
{c-user} systemctl status checkmk_relay-update-manager
● checkmk_relay-update-manager.service - Checkmk Relay Update Manager
     Loaded: loaded (/home/ubuntu/.config/systemd/user/checkmk_relay-update-manager.service; enabled; preset: enabled)
     Active: [.green]#activating (start)# since Tue 2026-01-20 17:47:38 CET; 46s ago
TriggeredBy: ● checkmk_relay-update-manager.path
   Main PID: 47108 (checkmk_relay-u)
      Tasks: 12 (limit: 9484)
     Memory: 772.8M (peak: 780.0M)
        CPU: 15.249s
     CGroup: /user.slice/user-1000.slice/user@1000.service/app.slice/checkmk_relay-update-manager.service
             ├─47108 /bin/bash /home/ubuntu/.local/bin/checkmk_relay-update-manager.sh
             └─47110 podman pull docker.io/checkmk/check-mk-relay:{current-major}p3

Here, the running `podman pull` process indicates that an update is running and the newest container is currently being downloaded.


[#restart]
=== Restarting the container

As hinted above, Podman tightly integrates with Systemd.
Thus, container restarts are simple restarts of Systemd units.

[{shell}]
----
{c-root} systemctl restart checkmk_relay
{c-root} systemctl status checkmk_relay
----
