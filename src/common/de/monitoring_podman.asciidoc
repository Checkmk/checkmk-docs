[#intro]
== Einleitung

{CMK} kann Container, die Sie mit Podman verwalten, über das xref:glossar#agent_plugin[Agentenplugin] `mk_podman.py` überwachen.
Dabei werden nicht nur die Container selbst, sondern auch Rahmendaten wie der Status Ihres Podman-Hosts überwacht.
Eine vollständige Liste der aktuell überwachbaren Dinge finden Sie wie immer im link:https://checkmk.com/integrations?tags=podman[Katalog der Check-Plugins^].

Das Agentenplugin kann auf zwei verschiedenen Wegen mit Podman kommunizieren:
Entweder über die API von Podman und somit über das zugehörige Unix-Socket oder direkt über das Kommando `podman`.
Wie Sie das Agentenplugin konfigurieren können, erfahren Sie in den Kapiteln xref:monitoring_podman#setup[Manuelle Einrichtung] bzw. xref:monitoring_podman#bakery[Einrichtung über die Agentenbäckerei].

Neben den Status- und Inventurinformationen, die {CMK} über den Podman-Host -- also den Host, auf dem Podman installiert ist und läuft -- ermitteln kann, kann {CMK} auch detaillierte Statusinformationen über die Container selbst ermitteln.
Hierzu wird in {CMK} jeder Container als eigenständiger Host angelegt, wenn er überwacht werden soll.
Seine Daten werden im xref:glossar#piggyback[Piggyback]-Verfahren mit zu Ihrem {CMK}-Server gebracht.

ifdef::onprem[]
In den kommerziellen Editionen 
endif::[]
ifdef::saas[]
In {CE}
endif::[]
können Sie mithilfe der xref:dcd#[dynamischen Host-Verwaltung] die passenden Hosts zu Ihren Containern auch automatisch anlegen und entfernen lassen.

[#install_agent_plugin]
=== Agentenplugin installieren

ifdef::onprem[]
Zusätzlich zum Agenten benötigen Sie noch das Agentenplugin `mk_podman.py`, das Sie in {CRE} über [.guihint]#Setup > Agents > Linux > Plug-ins# finden.
In den kommerziellen Editionen gelangen Sie im [.guihint]#Setup#-Menü über [.guihint]#Agents > Windows, Linux, Solaris, AIX# zunächst in die xref:wato_monitoringagents#bakery[Agentenbäckerei], wo Sie die gebackenen Pakete finden.
endif::[]
ifdef::saas[]
Zusätzlich zum Agenten benötigen Sie noch das Agentenplugin `mk_podman.py`.
In {CE} gelangen Sie im [.guihint]#Setup#-Menü über [.guihint]#Agents > Windows, Linux, Solaris, AIX# zunächst in die xref:wato_monitoringagents#bakery[Agentenbäckerei], wo Sie die gebackenen Pakete finden.
endif::[]
Von dort aus kommen Sie mit dem Menüeintrag [.guihint]#Related > Linux, Solaris, AIX files# zur Liste der Agentendateien.

Installieren Sie das Plugin auf dem Podman-Host in das xref:glossar#dir_plugins[Plugin-Verzeichnis] des Agenten (im Standardfall `/usr/lib/check_mk_agent/plugins`):

[{shell}]
----
{c-root} install -m 0755 mk_podman.py /usr/lib/check_mk_agent/plugins
----

Ausführlichere Informationen zur Installation eines Agentenplugins finden Sie im xref:agent_linux#plugins[Artikel zum Linux-Agenten].

Wenn Sie im Anschluss in {CMK} eine xref:wato_services#[Service-Erkennung] auf dem zugehörigen Host durchführen, sollten Sie bereits einige Services sehen, die das Wort Podman im Namen tragen:

.Ergebnis der Service-Erkennung auf dem Podman-Host
image::monitoring_podman_first_discovery.png[alt="Ansicht der neu in {CMK} gefundenen Podman-Services.",fullscreen=1]

Sollten Sie an dieser Stelle jedoch noch keine Podman-Services finden, dann kann dies daran liegen, dass die automatische Erkennung des Agentenplugins auf dem Host nicht auf den üblichen Wegen mit Podman kommunizieren konnte.
In diesem Fall müssen Sie dem Agentenplugin noch eine explizite Konfiguration mitgeben.
Wie das machen, erfahren Sie im folgenden Kapitel xref:monitoring_podman#configure[Agentenplugin konfigurieren].


[#configure]
=== Agentenplugin konfigurieren

Das Agentenplugin findet im Regelfall selbst heraus, wie es mit Podman auf dem jeweiligen Host kommunizieren kann.
Falls diese automatische Erkennung jedoch fehlschlägt oder Sie festlegen möchten, dass nur bestimmte Sockets oder beispielsweise nur das Root-Socket abgefragt werden sollen, dann können Sie das Plugin entsprechend konfigurieren.

Erstellen Sie dazu auf dem Podman-Host eine Konfigurationsdatei mit dem Namen `mk_podman.cfg` im xref:glossar#dir_conf[Konfigurationsverzeichnis] des Agenten.
Im Standardfall lautet dieses `/etc/check_mk/`.

ifdef::onprem[]
Eine Vorlage für diese Konfigurationsdatei mit ausführlichen Erläuterungen finden Sie als Instanzbenutzer Ihrer {CMK}-Instanz im Verzeichnis `~/share/check_mk/agents/cfg_examples/mk_podman.cfg`.
In der grafischen Oberfläche von {CRE} finden Sie die Beispielkonfiguration wieder über [.guihint]#Setup > Agents > Linux > Plug-ins#.
In den kommerziellen Editionen gelangen Sie im [.guihint]#Setup#-Menü über [.guihint]#Agents > Windows, Linux, Solaris, AIX# zunächst in die xref:wato_monitoringagents#bakery[Agentenbäckerei].
endif::[]
ifdef::saas[]
In {CE} finden Sie eine Vorlage für diese Konfigurationsdatei mit ausführlichen Erläuterungen.
Navigieren Sie im [.guihint]#Setup#-Menü über [.guihint]#Agents > Windows, Linux, Solaris, AIX# zunächst in die xref:wato_monitoringagents#bakery[Agentenbäckerei].
endif::[]
Von dort aus kommen Sie mit dem Menüeintrag [.guihint]#Related > Linux, Solaris, AIX files# zur Liste der Agentendateien.

Auf eine zentrale Einstellungsmöglichkeit möchten wir allerdings auch an dieser Stelle kurz eingehen.
Dabei handelt es sich um die `connection_method`, also die Art und Weise, wie `mk_podman.py` versucht, Ihre Podman-Instanz anzusprechen.
Standardmäßig versucht das Agentenplugin, über ein Unix-Socket mit der API von Podman zu sprechen.
Alternativ können Sie das Plugin anweisen, die Überwachung über das Kommando `podman` durchzuführen.
Was Sie hier verwenden müssen, hängt schlicht von Ihrer Podman-Umgebung ab.
Eine entsprechende Konfiguration dafür könnte so aussehen:

./etc/check_mk/mk_podman.cfg
[{file}]
----
[PODMAN]
connection_method: cli
piggyback_name_method: name_id
----

Nachdem Sie Ihre Konfiguration wie gewünscht angepasst haben, sind Sie bereit zum xref:#monitor_containers[Überwachen von Podman-Containern].

[#monitor_containers]
== Container überwachen

[#create_container_hosts]
=== Container-Hosts anlegen

Das eigentlich Interessante bei der Überwachung von Podman ist natürlich das Überwachen der Container.
Auch diese Daten besorgt das Agentenplugin.
Allerdings werden die Services nicht dem Podman-Host zugeordnet, sondern {CMK} bildet jeden Container als eigenen überwachbaren Host ab.

Die Zuordnung geschieht über den xref:glossar#piggyback[Piggyback]-Mechanismus.
Sie brauchen dann nur noch im Setup Hosts mit den richtigen Namen anzulegen und die Services werden diesen automatisch zugeordnet.

ifdef::onprem[]
In den kommerziellen Editionen können Sie diese Hosts automatisch anlegen lassen.
endif::[]
ifdef::saas[]
In {CE} können Sie diese Hosts automatisch anlegen lassen.
endif::[]
Verwenden Sie dazu in der xref:dcd#[dynamischen Host-Verwaltung] den Verbindungstyp [.guihint]#Piggyback data#.
Sie können die Hosts auch von Hand anlegen.

Beachten Sie dabei Folgendes:

* Der Host-Name muss exakt dem Verzeichnis entsprechen, welches in `~/tmp/check_mk/piggyback` angelegt wird.
* Falls die Container keine eigene IP-Adresse haben (was meist der Fall ist), stellen Sie im Abschnitt [.guihint]#Network address# das Attribut bei [.guihint]#IP address family# auf [.guihint]#No IP# ein.
* Bei [.guihint]#Monitoring agents# stellen Sie [.guihint]#{CMK} agent / API integrations# unbedingt auf [.guihint]#No API integrations, no {CMK} agent# ein.
* Das Feld [.guihint]#Parents# im Abschnitt [.guihint]#Basic settings# können Sie auf den Host-Namen des Podman-Hosts setzen.

Nachdem Sie die Container-Hosts angelegt und die Service-Erkennung durchgeführt haben, tauchen in dieser weitere Services auf.


[#rename_container_hosts]
=== Umbenennung der Container-Hosts

Die Namen für die Container-Hosts erzeugt bereits das Agentenplugin `mk_podman.py`.
Standardmäßig verknüpft das Plugin dafür den Namen des Podman-Hosts per Unterstrich mit dem Namen des Containers.
Abweichend davon können Sie das Agentenplugin anweisen, den Container-Namen mit der Container-ID zu kombinieren, oder tatsächlich nur den Container-Namen zu verwenden.
In der Konfigurationsdatei des Agentenplugin müssen Sie dafür die Option `piggyback_name_method` auf `name_id` beziehungsweise `name` setzen.
Wenn Sie sich für eine dieser beiden Methoden entscheiden, müssen Sie allerdings anderweitig sicherstellen, dass es in {CMK} nicht zu Namenskollisionen kommt.
In {CMK} kann es jeden Host-Namen nur genau einmal geben.

Falls Sie diese Namen dann noch nachträglich ändern wollen, um beispielsweise direkt am Host-Namen bestimmte Informationen ablesen zu können, steht Ihnen der Regelsatz [.guihint]#Host name translation for piggybacked hosts# zur Verfügung.
Damit können Sie recht flexibel Regeln zur Umbenennung von Host-Namen, die in Piggyback-Daten enthalten sind, festlegen.
In dem folgenden Beispiel wird allen Containern, die über den Host [.guihint]#mypodmanhost# Huckepack mitgebracht werden, die Zeichenkette `mypodmanhost-` vorangestellt.

image::monitoring_podman_hostname_translation.png[alt="Regel zur Umbenennung der in den Piggyback-Daten enthaltenen Host-Namen."]

Im Artikel xref:piggyback#renamehosts[Der Piggyback-Mechanismus] finden Sie weitere Möglichkeiten und eine genauere Beschreibung hierzu.


[#monitor_status_container_host]
=== Zustand der Container-Hosts überwachen

Im Auslieferungszustand enthält jede {CMK}-Instanz bereits eine Regel im Regelsatz [.guihint]#Host check command#, die dafür sorgt, dass jeder Container-*Host* den Zustand des zu diesem Host gehörigen Service [.guihint]#Status# erhält.
Diese Regel sollte unverändert bleiben bzw. genau so eingerichtet werden:

image::monitoring_podman_host_status.png[alt="Regel für den Befehl zur Überprüfung des Host-Zustand der Container."]

Der Service [.guihint]#Status# prüft eben, ob der Container läuft und kann daher als sicheres Mittel verwendet werden, um den Host-Zustand zu ermitteln.
Da jeder Container-Host automatisch das Host-Label `cmk/podman/object:container` zugewiesen bekommt, wird dieses hier als Bedingung verwendet.


[#diagnose]
== Diagnosemöglichkeiten

[#diagnose_podman_host]
=== Diagnose für einen Podman-Host

Sollte die Einrichtung nicht klappen, gibt es verschiedene Möglichkeiten der Analyse des Problems.

Prüfen Sie zuerst die Ausgabe des {CMK}-Agenten.
Diesen erhalten Sie in der Host-Ansicht im Monitoring über den Eintrag [.guihint]#Download agent output# des Aktionsmenüs.

image::monitoring_podman_download_agent_output.png[alt="Aktionsmenü des Hosts im Monitoring mit dem Eintrag zum Download der Agentenausgabe.", width=50%]

ifdef::onprem[]
Oder Sie durchsuchen als Instanzbenutzer direkt den Agent-Cache.

[{shell-raw}]
----
{c-omd} strings tmp/check_mk/cache/mypodmanhost | grep "<<<podman"
<<<podman_errors:sep(0)>>>
<<<podman_containers:sep(0)>>>
<<<podman_disk_usage:sep(0)>>>
<<<podman_engine:sep(0)>>>
<<<podman_pods:sep(0)>>>
<<<podman_container_inspect:sep(0)>>>
<<<podman_container_stats:sep(0)>>>
<<<podman_container_inspect:sep(0)>>>
<<<podman_container_stats:sep(0)>>>
----
endif::[]

Sollte die Ausgabe des Agenten keine Sektionen beinhalten, die mit `<<<podman` beginnen, müssen Sie auf dem Podman-Host weiter nach der Ursache suchen.

Führen Sie dafür auf dem Podman-Host das Agentenplugin manuell aus:

[{shell}]
----
{c-root} python3 /usr/lib/check_mk_agent/plugins/mk_podman.py
----

Die Ausgabe von `mk_podman.py` wird auch im Fehlerfall Hinweise darauf geben, was schief gelaufen ist.
Falls das Agentenplugin Podman auf dem Host beispielsweise nicht finden kann, dann wird es diesen Umstand unumwunden mitteilen:

[{shell}]
----
{c-root} python3 /usr/lib/check_mk_agent/plugins/mk_podman.py
mk_podman.py: Does not seem to be a podman host. Terminating.
----


[#diagnose_container_host]
=== Diagnose für einen Container-Host

Falls nur ein Container-Host keine Daten erhält bzw. keine Services für diesen erkannt werden, prüfen Sie zuerst, ob die Piggyback-Daten zu diesem Host vorhanden sind.
Prüfen Sie dazu erneut den Inhalt der Agentenausgabe dahingehend, ob Sektionen für den Container-Host vorhanden sind.
Sie können die Agentenausgabe beispielsweise nach der ID des Containers durchsuchen.

ifdef::onprem[]
Als Instanzbenutzer können Sie prüfen, ob zu einem Container Piggyback-Daten angelegt wurden und ob diese bereits in Ihrer {CMK}-Instanz verwendet werden.
Dies geschieht mit dem Kommandozeilenwerkzeug `cmk-piggyback`, welches wir im Artikel xref:piggyback#technology[Der Piggyback-Mechanismus] beschreiben.
endif::[]


[#host_labels]
== Host-Labels

{CMK} erzeugt auch für alle Podman-Objekte, die überwacht werden können, xref:labels#[Host-Labels].
Das Podman-Monitoring setzt unter anderem diese automatischen Labels: 

* für den Podman-Host das Label `cmk/podman/object:node`,
* für jeden Container die Labels `cmk/podman/object:container`, `cmk/podman/user:myuser` und `cmk/docker_image_name:myimage`,
* für alle Container, die Teil eines Pods sind, werden außerdem noch Labels erzeugt, welche den Pod identifizieren.

Diese Labels können Sie z.B. in Bedingungen für Ihre xref:wato_rules#[Regeln] verwenden, um Ihre Monitoring-Konfiguration abhängig von dem Image zu machen, das in einem Container verwendet wird.
