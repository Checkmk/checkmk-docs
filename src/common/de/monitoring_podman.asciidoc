[#intro]
== Einleitung

Podman ist eine daemonlose Container-Engine für Linux, die als Alternative zu Docker eingesetzt werden kann.
Anders als Docker benötigt Podman keinen zentralen Daemon und unterstützt rootless Container direkt.

{CMK} kann Podman-Hosts und die darauf laufenden Container überwachen – einschließlich Root- und Rootless-Container.
Die Überwachung erfolgt über das xref:glossar#agent_plugin[Agentenplugin] `mk_podman.py`, das auf dem Podman-Host installiert wird.
Jeder Container wird in {CMK} als eigenständiger Host abgebildet; seine Daten werden im xref:glossar#piggyback[Piggyback]-Verfahren an diesen Host geliefert.

ifdef::onprem[]
In den kommerziellen Editionen
endif::[]
ifdef::saas[]
In {CE}
endif::[]
können Sie mithilfe der xref:dcd#[dynamischen Host-Verwaltung] die Container-Hosts automatisch anlegen und entfernen lassen.

Eine vollständige Liste der verfügbaren Check-Plugins finden Sie im link:https://checkmk.com/de/integrations?tags=podman[Katalog der Check-Plugins^].


[#prerequisites]
== Voraussetzungen

Für das Podman-Monitoring gelten folgende Voraussetzungen:

* Auf dem Podman-Host ist der xref:agent_linux#[Linux-Agent] von {CMK} installiert und aktiv.
* Das Python-Paket `requests` ist auf dem Podman-Host installiert (siehe xref:python_requests[Python-Paket installieren]).
* Der Podman-Socket-Dienst ist aktiv, wenn die Standard-Verbindungsmethode (API) verwendet wird (siehe xref:socket_activation[Socket aktivieren]).

[#setup]
== Einrichtung

[#install_agent_plugin]
=== Agent und Plugin installieren

Damit Sie einen Podman-Host überwachen können, muss dieser zunächst mit dem xref:agent_linux#[Linux-Agenten] überwacht werden.
Dadurch erhalten Sie ein Basis-Monitoring des Wirtssystems, jedoch noch keine Informationen über Podman oder die Container.

Dazu benötigen Sie das xref:glossar#agent_plugin[Agentenplugin] `mk_podman.py`, das Sie in {CMK} unter [.guihint]#Setup > Agents > Other operating systems > Plugins# finden.

Installieren Sie das Plugin in das Plugin-Verzeichnis des Agenten (in der Regel `/usr/lib/check_mk_agent/plugins`):

[{shell}]
----
{c-root} install -m 0755 mk_podman.py /usr/lib/check_mk_agent/plugins
----

ifdef::onprem[]
In den kommerziellen Editionen
endif::[]
ifdef::saas[]
In {CE}
endif::[]
können Sie alternativ die xref:wato_monitoringagents#bakery[Agentenbäckerei] nutzen.
Den zugehörigen Regelsatz finden Sie unter [.guihint]#Setup > Agents > Windows, Linux, Solaris, AIX > Agent rules > Podman hosts and containers (Linux)#.
Aktivieren Sie dort [.guihint]#Deploy Podman plug-in# und backen Sie anschließend den Agenten.
Nach der automatischen Installation des neuen Agentpakets auf dem Host können Sie direkt mit der xref:wato_services#[Service-Erkennung] fortfahren.


[#socket_activation]
=== Socket aktivieren

Das Agentenplugin kommuniziert standardmäßig über einen Unix-Socket mit der Podman-API.
Der entsprechende systemd-Dienst muss daher aktiv sein.

Für den Root-Socket aktivieren Sie den Dienst als Root:

[{shell}]
----
{c-root} systemctl enable --now podman.socket
----

Für Rootless-Container aktivieren Sie den Socket für den jeweiligen Benutzer:

[{shell}]
----
{c-user} systemctl --user enable --now podman.socket
----

Im Standard-Modus (`auto`) erkennt das Plugin automatisch sowohl den Root-Socket unter `/run/podman/podman.sock` als auch Benutzer-Sockets nach dem Muster `/run/user/{uid}/podman/podman.sock`.

[TIP]
====
Als Alternative zur API-Verbindung über Sockets können Sie das Plugin mit der CLI-Verbindungsmethode betreiben, die den `podman`-Befehl direkt aufruft.
Diese Methode ist dann nützlich, wenn der Podman-Socket-Dienst nicht verfügbar ist.
Konfigurieren Sie dazu `connection_method: cli` in der Konfigurationsdatei (siehe xref:finetuning[Plugin feinjustieren]).
====


[#python_requests]
=== Python-Paket installieren

Das Agentenplugin benötigt das Python-Paket `requests`.
Prüfen Sie dessen Vorhandensein auf der Befehlszeile des Podman-Hosts:

[{shell}]
----
{c-root} python3 -c "import requests; print(requests.__version__)"
2.28.2
----

Falls das Paket nicht installiert ist, installieren Sie es bevorzugt über das Paketmanagement Ihrer Linux-Distribution.
Auf RHEL- und Rocky-Linux-Systemen finden Sie das Paket `python3-requests` im EPEL-Repository:

[{shell}]
----
{c-root} dnf install python3-requests
----

[NOTE]
====
Bei der Installation mit `pip3` besteht die Gefahr, von der Distribution mitgelieferte Python-Pakete zu beschädigen.
Nutzen Sie daher bevorzugt das Paketmanagement Ihrer Distribution.
====


[#finetuning]
=== Plugin feinjustieren

Sie können das Verhalten des Plugins über eine Konfigurationsdatei anpassen.
Erstellen Sie dazu auf dem Podman-Host die Datei `/etc/check_mk/mk_podman.cfg`.
Eine vollständige kommentierte Vorlage finden Sie im {CMK}-Verzeichnis unter `~/share/check_mk/agents/cfg_examples/mk_podman.cfg`.

Die wichtigsten Einstellungen im Überblick:

[cols="30,~"]
|===
|Option |Beschreibung

|`connection_method` |Verbindungsmethode: `api` (Standard, über Unix-Socket) oder `cli` (Podman-Befehlszeile ohne Socket)
|`socket_detection_method` |Socket-Erkennung: `auto` (Standard), `only_root_socket`, `only_user_sockets` oder `manual`
|`socket_paths` |Manuelle Socket-Pfade, kommagetrennt – nur bei `socket_detection_method: manual`
|`piggyback_name_method` |Namensbildung der Piggyback-Hosts (siehe xref:piggyback_naming[Piggyback-Hostnamen konfigurieren])
|===

ifdef::onprem[]
In den kommerziellen Editionen
endif::[]
ifdef::saas[]
In {CE}
endif::[]
lassen sich alle Parameter komfortabel in der xref:wato_monitoringagents#bakery[Agentenbäckerei] über den Regelsatz [.guihint]#Podman hosts and containers (Linux)# konfigurieren.

Nach der Installation und Konfiguration des Plugins führen Sie in {CMK} eine xref:wato_services#[Service-Erkennung] für den Podman-Host durch.
Auf dem Podman-Host erscheinen anschließend unter anderem die folgenden Services:

* [.guihint]#Podman status# – zeigt Fehler des Plugins an
* [.guihint]#Podman containers# – Anzahl der Container nach Zustand
* [.guihint]#Podman disk usage: images#, [.guihint]#... containers#, [.guihint]#... volumes# – Festplattennutzung nach Typ
* [.guihint]#Podman pods# – Anzahl der Pods nach Zustand


[#containers]
=== Container überwachen

==== Container-Hosts anlegen

Das Agentenplugin liefert die Daten der Container im xref:glossar#piggyback[Piggyback]-Verfahren.
{CMK} legt diese Daten im Verzeichnis `~/tmp/check_mk/piggyback` ab.
Damit die Services der Container im Monitoring erscheinen, muss für jeden Container ein eigener Host in {CMK} angelegt werden.

Beachten Sie beim manuellen Anlegen Folgendes:

* Der Host-Name muss exakt dem Verzeichnisnamen in `~/tmp/check_mk/piggyback` entsprechen.
  In der Standardeinstellung lautet er `{nodename}_{containername}`, zum Beispiel `mypodmanhost_nginx`.
* Falls der Container keine eigene IP-Adresse hat (was in der Regel zutrifft), stellen Sie im Abschnitt [.guihint]#Network address# das Attribut [.guihint]#IP address family# auf [.guihint]#No IP# ein.
* Bei [.guihint]#Monitoring agents# setzen Sie [.guihint]#{CMK} agent / API integrations# auf [.guihint]#No API integrations, no {CMK} agent#.
* Im Abschnitt [.guihint]#Basic settings# können Sie das Feld [.guihint]#Parents# auf den Namen des Podman-Hosts setzen.
* Der Podman-Host und seine Container müssen von derselben {CMK}-Instanz überwacht werden.

ifdef::onprem[]
In den kommerziellen Editionen
endif::[]
ifdef::saas[]
In {CE}
endif::[]
können Sie Container-Hosts auch automatisch anlegen lassen.
Verwenden Sie dazu in der xref:dcd#[dynamischen Host-Verwaltung] den Verbindungstyp [.guihint]#Piggyback data#.

Nach der Service-Erkennung für die Container-Hosts erscheinen dort unter anderem die folgenden Services:

* [.guihint]#Status# – Container-Zustand (z.B. `Running`, `Exited`, `Paused`)
* [.guihint]#CPU utilization# – CPU-Auslastung des Containers
* [.guihint]#Memory# – Arbeitsspeichernutzung des Containers
* [.guihint]#Disk I/O# – Festplattenaktivität des Containers
* [.guihint]#Health# – Ergebnis des im Container konfigurierten Health-Checks (sofern vorhanden)
* [.guihint]#Restart count# – Anzahl der Neustarts des Containers
* [.guihint]#Uptime# – Laufzeit des Containers seit dem letzten Start


[#piggyback_naming]
==== Piggyback-Hostnamen konfigurieren

In der Standardeinstellung (`nodename_name`) wird jeder Container-Host nach dem Schema `{nodename}_{containername}` benannt, zum Beispiel `mypodmanhost_nginx`.
Dies verhindert Namenskollisionen, wenn auf verschiedenen Podman-Hosts Container mit identischem Namen laufen.

Die Namensbildung lässt sich über die Option `piggyback_name_method` in der Konfigurationsdatei `/etc/check_mk/mk_podman.cfg` anpassen:

[cols="25,~"]
|===
|Wert |Beschreibung

|`nodename_name` (Standard) |`{nodename}_{containername}` – empfohlen, da Kollisionen bei gleichnamigen Containern auf verschiedenen Hosts vermieden werden
|`name` |Nur der Container-Name – stabil, aber Kollisionen bei gleichnamigen Containern möglich
|`name_id` |`{containername}_{container-id}` – eindeutig, jedoch kann sich die Container-ID in manchen Fällen ändern
|===

ifdef::onprem[]
In den kommerziellen Editionen
endif::[]
ifdef::saas[]
In {CE}
endif::[]
konfigurieren Sie dies in der xref:wato_monitoringagents#bakery[Agentenbäckerei] unter [.guihint]#Podman hosts and containers (Linux) > Host name used for containers#.

Über den Regelsatz [.guihint]#Host name translation for piggybacked hosts# können Sie Hostnamen aus Piggyback-Daten außerdem flexibel umbenennen.
Weitere Details hierzu finden Sie im Artikel xref:piggyback#renamehosts[Der Piggyback-Mechanismus].


==== Host-Zustand überwachen

ifdef::onprem[]
Da der xref:monitoring_basics#hosts[Host-Zustand] eines Containers nicht über xref:cmc_differences#smartping[TCP-Pakete oder ICMP] ermittelt werden kann, bietet sich der Service [.guihint]#Status# des jeweiligen Containers an.
endif::[]
ifdef::saas[]
Für die Überwachung des xref:monitoring_basics#hosts[Host-Zustandes] eines Containers bietet sich der Service [.guihint]#Status# des jeweiligen Containers an.
endif::[]
Dieser prüft, ob der Container läuft, und kann daher als zuverlässige Grundlage für den Host-Zustand dienen.
Legen Sie dazu eine Regel im Regelsatz [.guihint]#Host check command# an, setzen Sie [.guihint]#Use the status of the service...# auf den Service [.guihint]#Status#, und schränken Sie die Bedingungen so ein, dass die Regel nur Container-Hosts betrifft.


[#diagnose]
== Diagnosemöglichkeiten

=== Diagnose für den Podman-Host

Sollte die Einrichtung nicht wie erwartet funktionieren, prüfen Sie zunächst, ob das Plugin Daten liefert.
Sie können die Agentenausgabe als Textdatei herunterladen: Öffnen Sie dazu in einer Host-Ansicht im Monitoring das Aktionsmenü und wählen Sie [.guihint]#Download agent output#.

ifdef::onprem[]
Alternativ können Sie den Agenten-Cache direkt auf dem {CMK}-Server durchsuchen:

[{shell-raw}]
----
{c-omd} strings tmp/check_mk/cache/mypodmanhost | grep "<<<podman"
<<<podman_errors:sep(0)>>>
<<<podman_containers:sep(0)>>>
<<<podman_disk_usage:sep(0)>>>
<<<podman_engine:sep(0)>>>
<<<podman_pods:sep(0)>>>
----
endif::[]

Falls diese Sektionen fehlen oder der Service [.guihint]#Podman status# den Zustand {CRIT} anzeigt, prüfen Sie Folgendes:

* Ist das Plugin vorhanden und ausführbar? Prüfen Sie mit `ls -l /usr/lib/check_mk_agent/plugins/mk_podman.py`.
* Ist der Podman-Socket-Dienst aktiv? Prüfen Sie mit `systemctl status podman.socket`.
* Ist das Python-Paket `requests` installiert? Prüfen Sie mit `python3 -c "import requests"`.
* Erkennt das Plugin den Host als Podman-Host? Es prüft auf `/var/lib/podman`, `/run/podman` oder den Befehl `podman`.

Die Fehlermeldung des Service [.guihint]#Podman status# enthält eine genauere Beschreibung des Problems, beispielsweise den genauen Pfad des nicht gefundenen Sockets.


=== Diagnose für Container-Hosts

Falls ein Container-Host keine Daten erhält, prüfen Sie zunächst, ob die Piggyback-Daten im richtigen Verzeichnis abgelegt werden.
Der Host-Name in {CMK} muss exakt dem Verzeichnisnamen in `~/tmp/check_mk/piggyback` entsprechen.

ifdef::onprem[]
[{shell}]
----
{c-omd} ls tmp/check_mk/piggyback/
mypodmanhost_nginx  mypodmanhost_myapp
----
endif::[]

Falls der Name nicht übereinstimmt, können Sie im Regelsatz [.guihint]#Host name translation for piggybacked hosts# eine explizite Zuordnung per [.guihint]#Explicit hostname mapping# vornehmen.


[#host_labels]
== Host-Labels

Das Podman-Monitoring setzt automatisch xref:labels#[Host-Labels], die Sie als Bedingungen in xref:wato_rules#[Regeln] nutzen können:

Für den Podman-Host (Node):

* `cmk/podman/object:node` – kennzeichnet den Host als Podman-Node
* `cmk/podman/host:{hostname}` – Hostname des Podman-Nodes

Für jeden Container:

* `cmk/podman/object:container` – kennzeichnet den Host als Podman-Container
* `cmk/podman/user:{user}` – Benutzer, dem der verwendete Podman-Socket gehört
* `cmk/podman/pod:{pod}` – Pod, in dem der Container läuft (nur gesetzt, wenn der Container Teil eines Pods ist)
* `cmk/docker_image` – vollständiger Image-Name, z.B. `docker.io/library/nginx:latest`
* `cmk/docker_image_name` – Image-Name ohne Registry und Tag, z.B. `nginx`
* `cmk/docker_image_version` – Image-Tag, z.B. `latest`

[NOTE]
====
Die Labels `cmk/docker_image`, `cmk/docker_image_name` und `cmk/docker_image_version` sind absichtlich identisch zu den Labels des xref:monitoring_docker#host_labels[Docker-Monitorings].
So können Sie Docker- und Podman-Container mit denselben Regeln verwalten.
====


[#inventory]
== HW/SW-Inventur

Das Podman-Monitoring liefert Inventurdaten, die Sie in der xref:inventory#[HW/SW-Inventur] einsehen können.

Für den Podman-Host enthält die Inventur unter [.guihint]#Software > Applications > Podman# unter anderem:

* Betriebsmodus: Rootful oder Rootless
* API-Version
* Anzahl der Container nach Zustand (laufend, pausiert, gestoppt, beendet)
* Anzahl der Images

Eine Tabelle aller Container mit ID, Name, Status, Image, Labels und Erstellungszeitpunkt sowie eine Tabelle aller Images mit ID, Repository, Tag und Größe sind ebenfalls abrufbar.

Für jeden Container enthält die Inventur außerdem:

* Hostname, Pod-Zugehörigkeit und Labels
* Netzwerkinformationen: IP-Adresse, Gateway, MAC-Adresse
