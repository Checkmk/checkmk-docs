1. Termes spécifiques à Checkmk

Active check

Un check actif est un petit programme ou script qui établit une connexion directe avec un service sur le réseau ou internet, à partir duquel il interroge les données de supervision. Les contrôles actifs sont utilisés pour les services réseau tels que HTTP, SMTP ou IMAP, par exemple check_http pour interroger des pages web. Un contrôle actif gère à la fois la collection et l'évaluation des données. Cela diffère d'un plugin de supervision, parfois appelé contrôle passif, car un contrôle passif évalue uniquement les données existantes.

Pour en savoir plus, consultez la section Supervision des services réseau (vérifications actives).

Activer les modifications

Les modifications apportées à la configuration ne prennent effet sur la supervision qu'après avoir été activées dans une deuxième étape, de manière similaire à la façon dont les programmes de partitionnement gèrent cela, par exemple : Configurer, vérifier, appliquer.

Pour plus d'informations, veuillez consulter la section Activer les modifications.

Agent

Un agent sur un ordinateur hôte collecte les données pertinentes pour la supervision. L'agent peut être un petit programme installé sur l'ordinateur hôte (l'agent Checkmk), un agent SNMP fonctionnant indépendamment de Checkmk sur l'ordinateur hôte, un agent spécial qui obtient les informations via une API fournie par le système cible ou un contrôle actif qui interroge les services réseau.

Pour en savoir plus, consultez la section Agents de supervision.

Agent spécial

Sur certains systèmes, l'agent Checkmk standard ne peut pas être installé et SNMP n'est pas disponible (de manière satisfaisante). À la place, ces systèmes fournissent des API de gestion basées sur Telnet, SSH ou HTTP/XML. Via un agent spécial exécuté sur le serveur Checkmk, Checkmk interroge ces interfaces et intègre l'ordinateur hôte dans Checkmk via l'API.

Pour plus d'informations, veuillez consulter la section Agents spéciaux.

Appliance virtuelle

L'appliance virtuelle est un système créé pour VirtualBox ou VMware ESXi avec un micrologiciel préinstallé préparé pour Checkmk. Elle comprend une interface de configuration graphique qui élimine tout besoin de connaissances en Linux.

Pour en savoir plus, consultez la section Appliance virtuelle.

Balise de l'hôte

Les balises de l'hôte sont des caractéristiques qui peuvent être attribuées aux hôtes afin de pouvoir les cibler ultérieurement dans la configuration des règles ou dans la supervision pour les vues. Les balises de l'hôte sont divisées en groupes, par exemple un groupe de balises. Les systèmes d'exploitation peuvent être configurés avec les balises Linux et Windows. Certains groupes de balises sont prédéfinis, tels que le type d'agent Checkmk utilisé ou la famille d'adresses IP utilisée pour enregistrer si un hôte doit être supervisé via IPv4, IPv6 ou les deux versions. Les balises ont également des valeurs prédéfinies et une valeur par défaut qui est attribuée à chaque hôte tant qu'elle n'a pas été remplacée par une autre option du groupe.

Pour en savoir plus, consultez la section Balises de l'hôte.

Barre de navigation

La barre de navigation est le panneau de navigation principal de l'interface Checkmk. Elle se trouve à gauche et contient notamment les menus « Monitor, » (Configuration), « Setup » (Problèmes) et « Customize » (Problèmes).

Pour plus d'informations, veuillez consulter la section La barre de navigation.

Barre latérale

La barre latérale peut être affichée à partir de la barre de navigation d'un simple clic de souris. Les utilisateurs peuvent ajouter divers snap-ins à la barre latérale pour faciliter la navigation ou afficher d'un coup d'œil les données d'état importantes.

Pour plus d'informations, veuillez consulter la section Barre latérale.

boulangerie d’agents

Dans les éditions commerciales, la boulangerie d'agents est utilisée pour emballer individuellement les agents et, en option, les paquets peuvent également être distribués automatiquement.

Pour en savoir plus, consultez la section Boulangerie d'agents.

Check

Dans Checkmk, un check est un script ou un programme qui évalue un ordinateur hôte ou un service selon des règles prédéfinies, c'est-à-dire le processus qui détermine le statut des ordinateurs hôtes et des services et qui aboutit à l'affichage d'un statut d'OK, DOWN, UNREACH, WARN, CRIT, PEND ou UNKNOWN. Les checks peuvent être mis en œuvre, par exemple, avec un plugin de supervision, un check local ou un check actif.

Pour plus d'informations, veuillez consulter la rubrique Vérifications.

Check local

Un check local est une extension (écrite par l'utilisateur) sous forme de script dans n'importe quel langage de programmation, qui s'exécute sur l'ordinateur hôte surveillé. Contrairement aux checks réguliers, le calcul du statut s'effectue directement sur l'ordinateur hôte. Les résultats d'un script sont ajoutés à la sortie régulière de l'agent.

Plus d'informations sous Checks locaux.

Configuration rapide

Une configuration rapide comprend quatre composants que vous connaissez déjà individuellement depuis la configuration d'une instance Checkmk. Il s'agit d'une règle pour l'agent spécial correspondant, d'un hôte, d'une connexion dans la gestion dynamique des hôtes et d'une entrée dans le coffre-fort à mot de passe.

Contact

Les contacts sont des utilisateurs Checkmk responsables de hôtes et de services spécifiques. L'affectation des contacts aux hôtes et aux services s'effectue via des groupes de contacts. Les contacts peuvent également être des comptes utilisateurs qui existent uniquement pour les notifications, par exemple pour le transfert vers un système de tickets.

Pour plus d'informations, consultez la section Groupes de contacts.

Édition

Les éditions Checkmk sont les différentes variantes logicielles de Checkmk disponibles au téléchargement et à l'installation. Il s'agit de la version libre Checkmk Raw, de la version Checkmk Enterprise disponible sur souscription pour les utilisateurs professionnels, de la version Checkmk Cloud basée sur Enterprise et de la version multitenant Checkmk MSP. Dans cet ordre, chaque édition contient l'ensemble des fonctions de l'édition précédente et les complète par des fonctions supplémentaires spécifiques. Par exemple, Checkmk Cloud est basé sur Checkmk Enterprise, amélioré et optimisé pour les environnements cloud.

Pour en savoir plus, consultez la section Sélectionner une édition Checkmk.

Environnement de configuration

L'interface web Checkmk est divisée en environnements de supervision et de configuration. Ce dernier fait référence aux processus permettant de définir des règles, d'ajouter et de définir des hôtes et des services, de gérer les utilisateurs ou de spécifier des options générales. L'environnement de configuration est accessible via le menu « Setup » dans la barre de navigation.

Pour plus d'informations, consultez la section Interface utilisateur.

Environnement de supervision

L'interface web Checkmk est divisée en environnements de supervision et de configuration. Le premier fait référence aux processus par lesquels l'état de l'infrastructure supervisée est affiché ; il s'agit notamment de l'inventaire, des tableaux de bord, des listes d'hôtes, de services, d'événements ou de problèmes, des données historiques, etc. L'environnement de supervision est accessible via le menu « Monitor » de la barre de navigation.

Plus d'informations sous L'interface utilisateur.

État de l'ordinateur hôte

L'état d'un ordinateur hôte, c'est-à-dire s'il est accessible via le réseau (UP), s'il ne répond pas aux requêtes du réseau (DOWN) ou si son chemin d'accès est bloqué par des périphériques intermédiaires défectueux (commutateurs, routeurs, etc.) (UNREACH). Pour les ordinateurs hôtes récemment ajoutés qui n'ont jamais été interrogés auparavant, il existe également l'état PEND, qui n'est pas un état au sens propre du terme.

Pour plus d'informations, veuillez consulter la section Ordinateurs hôtes et services.

État du service

L'état d'un service est toujours « OK » (en cours), « WARN » (en attente), « CRIT » (en cours de démarrage) ou « UNKNOWN » (inconnu) et décrit l'état actuel du service selon des règles prédéfinies. Pour les services récemment ajoutés qui n'ont jamais été interrogés auparavant, il existe également l'état « PEND », qui n'est pas un état au sens strict.

Pour plus d'informations, veuillez consulter la section Services.

Étiquette

Les hôtes peuvent recevoir des balises d'hôte, mais ils peuvent également recevoir des étiquettes directes. Une étiquette se compose de deux parties (clé et valeur), séparées par deux points. Ces paires clé-valeur arbitraires (os:linux, os:windows, foo:bar, etc.) peuvent être définies directement sur un hôte, sans configuration préalable, contrairement aux balises de l'hôte. Les étiquettes n'ont donc pas de portée prédéfinie, ni de valeur par défaut, mais sont très dynamiques. En particulier, Checkmk peut prendre en charge la supervision d'objets créés automatiquement par des systèmes de conteneurs tels que Kubernetes, Azure ou AWS en tant qu'ordinateurs hôtes, puis les enrichir avec des étiquettes générées automatiquement à partir de leurs métadonnées. Les étiquettes peuvent être utilisées, par exemple, pour sélectionner des conditions dans des règles ou pour filtrer dans les vues de supervision.

Pour en savoir plus, consultez la section Étiquettes.

Event Console (EC)

Lors de la supervision des ordinateurs hôtes et des services, Checkmk se concentre sur les états. La console d'événements est le module qui, en revanche, s'occupe des événements, c'est-à-dire de la supervision à partir de sources telles que syslog ou les traps SNMP, mais aussi, en option, du journal des événements Windows, des fichiers journaux et des applications propres. Exemple : un message d'avertissement du service SMTP sur un serveur de messagerie ne modifierait pas le statut/l'état de son hôte ou de ses services, mais il s'agit néanmoins d'une information pertinente qui doit être prise en compte dans la supervision. La console d'événements permet d'enregistrer et d'afficher de tels événements dans Checkmk.

Pour en savoir plus, consultez la section Console d'événements.

Ferroutage

Certains hôtes faisant partie de la supervision ne sont pas interrogés directement car il ne s'agit pas d'appareils physiques, mais plutôt de machines virtuelles ou de conteneurs, ou encore parce que les données ne peuvent être fournies que par un système tiers. Ces systèmes tiers (les hôtes physiques) fournissent les données sous forme de pièce jointe dans leur propre sortie d'agent et sont appelés hôtes ferroutés par nos soins. Ainsi, par exemple, un serveur Docker ferrouterait les données du conteneur avec ses propres données. Les conteneurs de cet exemple sont alors les hôtes ferroutés.

Pour en savoir plus, consultez la section Le mécanisme de ferroutage.

Groupe d'hôtes

Dans Checkmk, les hôtes sont principalement gérés par des dossiers. Les groupes d'hôtes permettent de regrouper différemment les hôtes afin de sélectionner des hôtes pour la supervision, par exemple dans les vues de la table. Les balises de l'hôte, les étiquettes et les dossiers sont utilisés pour affecter les hôtes à ces groupes via des règles. Les hôtes peuvent également être affectés explicitement à un groupe d'hôtes.

Pour plus d'informations, veuillez consulter la section Groupes d'hôtes.

Groupe de services

Tout comme les hôtes, les services peuvent également être regroupés afin que ces groupes puissent être filtrés ultérieurement dans les vues ou traités spécifiquement dans la configuration. Les groupes peuvent être formés à partir de dossiers, de balises d’hôtes, d’étiquettes d’hôtes et de services, ainsi que de noms d’hôtes et de services filtrés via des expressions régulières.

Pour plus d'informations, consultez la section Groupes de services.

Informatique décisionnelle (BI)

L'informatique décisionnelle dans Checkmk permet d'afficher clairement l'état global d'un aperçu de niveau supérieur dérivé de nombreuses valeurs d'état individuelles. Ce niveau supérieur peut être un regroupement abstrait de composants individuels ou une application critique pour l'entreprise. Par exemple, l'état d'une application de messagerie électronique, composée de différents hôtes, commutateurs et services tels que SMTP et IMAP, peut être combiné en un seul affichage. Il est également possible de formuler des considérations totalement intangibles et non techniques, par exemple la disponibilité ponctuelle d'un produit à livrer : cet objectif se situe dans le futur et dépend de nombreux facteurs, tels que la chaîne d'approvisionnement, le bon fonctionnement de la production, le personnel disponible, etc. Toute menace pesant sur cet objectif abstrait peut être identifiée et enregistrée via le module BI.

Pour en savoir plus, consultez la rubrique Informatique décisionnelle (BI).

Instable

Si un objet (ordinateur hôte ou service) change d'état plusieurs fois en peu de temps, il est considéré comme instable et marqué de l'icône . Pour un objet identifié comme instable, les changements d'état successifs ne déclenchent aucune notification supplémentaire. La détection d'instabilité peut être activée ou désactivée, globalement ou par règle, séparément pour les ordinateurs hôtes et les services.

Pour plus d'informations, veuillez consulter la section Ordinateurs hôtes et services instables.

Instance

Une instance est un projet de supervision Checkmk unique et actuellement en cours d'exécution. Checkmk peut être exécuté en parallèle sur le même serveur sous forme d'instances multiples et indépendantes, par exemple pour tester différentes versions ou éditions de Checkmk, ou pour exécuter une supervision distincte pour de (nouveaux) ordinateurs hôtes qui ne doivent pas (encore) être inclus dans la supervision de production.

Pour plus d'informations, consultez la section Création d'une instance.

Intégrations API

Lorsque la configuration Checkmk fait référence à l'API integrations, il s'agit de données de supervision qui utilisent le format de données de l'agent Checkmk, mais qui proviennent d'une source différente. Ces sources peuvent être des programmes sources de données, des agents spéciaux ou des hôtes qui ferroutent leurs données. Si les données reçues via une intégration API doivent être utilisées dans la supervision, l'option « API integrations » doit être activée dans les propriétés d'un hôte.

Pour plus d'informations, veuillez consulter la section Programmes de source de données.

Jeu de règles

CPU utilization on Linux/Unix Un jeu de règles s'applique à un aspect spécifique d'un hôte ou d'un service, tel que les valeurs seuils d'utilisation du processeur. Vous pouvez créer autant de règles individuelles que vous le souhaitez dans chaque jeu de règles. Par exemple, le jeu de règles « » (Utilisation du processeur) pourrait contenir deux règles qui définissent le statut du service sur « WARN » (Avertissement) à 90 % sur certains hôtes et à 70 % sur d'autres.

Pour plus d'informations, consultez la section Types de jeux de règles.

Livestatus

Livestatus est l'interface la plus importante de Checkmk. Elle permet aux utilisateurs de Checkmk d'accéder en temps réel et le plus rapidement possible à toutes les données des hôtes et des services surveillés. Par exemple, les données du snap-in « Overview » sont récupérées directement via cette interface. Le fait que les données soient récupérées directement depuis la RAM évite les accès lents au disque et permet un accès rapide à la supervision sans surcharger le système.

Pour en savoir plus, consultez la section Récupération des données d'état via Livestatus.

Métrique

Valeurs mesurables et calculables pour les ordinateurs hôtes et les services au fil du temps, telles que la température, l'utilisation ou la disponibilité, qui peuvent être utilisées pour des graphiques, par exemple. Les valeurs passées sont stockées dans des RRD (Round Robin Database) et conservées par défaut pendant 4 ans maximum.

Pour plus d'informations, consultez la section Valeurs mesurées et graphiques.

Mise à jour de l’agent

Dans les éditions commerciales, la mise à jour de l’agent est un plugin d’agent qui permet de mettre à jour automatiquement les agents.

Pour plus d'informations, veuillez consulter la section Mises à jour automatiques des agents.

Mode Pull

En mode Pull, l'agent Checkmk écoute les connexions entrantes provenant du serveur Checkmk sur le port TCP 6556. Dès que l'agent reçoit une requête, il envoie les données de supervision au serveur. Ici, l'initiative de la transmission des données vient du serveur, qui extrait en quelque sorte les données de l'agent. Le mode Pull est le mode de transmission par défaut des données de l'agent Checkmk et fonctionne dans toutes les éditions Checkmk.

Pour en savoir plus, consultez la section Agent Checkmk.

Mode Push

En mode Push, l'agent Checkmk envoie les données de supervision au serveur Checkmk une fois par minute. L'agent déclenche lui-même la transmission des données et n'attend pas de requête du serveur. Le mode Push est toujours nécessaire si le serveur Checkmk ne peut pas accéder au réseau dans lequel se trouvent l'ordinateur hôte à superviser et son agent, par exemple dans une configuration basée sur le cloud. Pour cette raison, le mode Push n'est disponible qu'à partir de Checkmk Cloud, c'est-à-dire dans Checkmk Cloud et Checkmk MSP.

Pour en savoir plus, consultez la section Agent Checkmk.

Notification

Une notification permet d'informer activement un utilisateur Checkmk de problèmes ou d'autres événements de supervision, par courrier électronique HTML, SMS, Slack ou similaire. Les règles de notification déterminent qui est notifié et comment. Par exemple, si M. Hirsch reçoit un courrier électronique l'informant que le service filesystem / sur l'ordinateur hôte myserver123 est passé de WARN à CRIT, c'est parce que M. Hirsch est un contact pour cet ordinateur hôte et qu'une règle de notification stipule que tous les contacts de l'ordinateur hôte doivent recevoir un courrier électronique lorsque l'un de ses services passe à CRIT.

Pour plus d'informations, veuillez consulter la section Notifications de base.

Ordinateur hôte

Dans Checkmk, un ordinateur hôte est tout système autonome, physique ou virtuel, supervisé par Checkmk. Il s'agit généralement de composants disposant de leur propre adresse IP (serveurs, commutateurs, périphériques SNMP, machines virtuelles), mais aussi, par exemple, de conteneurs Docker ou d'autres objets logiques qui ne disposent pas d'une telle adresse IP. Chaque ordinateur hôte a toujours l'un des statuts suivants : UP, DOWN, UNREACH ou PEND, et dispose toujours d'au moins un service.

Pour aller plus loin : Pour Checkmk, en interne, un hôte est simplement un élément de structuration qui contient des éléments à surveiller, c'est-à-dire des services. Chaque hôte dispose nécessairement d'au moins un service pour vérifier l'accessibilité réelle (tel que PING ou l'agent Checkmk lui-même, c'est-à-dire le service Check_MK). À cet égard, le terme « hôte » ne signifie guère plus que l'en-tête sous lequel un certain nombre de services sont regroupés.

Pour plus d'informations, veuillez consulter la section Administration des ordinateurs hôtes.

pack d'extension Checkmk (MKP)

MKP est le format de fichier propre à Checkmk pour l'agrégation et la distribution d'extensions, c'est-à-dire de plugins de supervision personnalisés, de plugins d'agent, de définitions de visualisation de la série chronologique, de scripts de notification, de vues, de tableaux de bord, etc.

Pour en savoir plus, consultez la section Packages d'extension Checkmk (MKP).

Période de temps

Dans Checkmk, il est possible de limiter certaines actions telles que les notifications, les calculs de disponibilité et même l'exécution générale des checks à certaines périodes récurrentes. Les périodes peuvent être utilisées, par exemple, pour définir les heures de travail quotidiennes, pour spécifier les congés et les jours fériés, ou pour séparer les week-ends des jours ouvrables. Ces périodes peuvent ensuite être utilisées dans les règles.

Pour plus d'informations, consultez la section Périodes de temps.

Périodes de maintenance planifiées

Les périodes de maintenance planifiées sont des interruptions planifiées, par exemple pour la mise à jour de certains hôtes. Les périodes de maintenance planifiées remplacent temporairement les notifications et sont, par exemple, prises en compte dans le calcul de la disponibilité et empêchent temporairement les hôtes et services associés d'apparaître comme des problèmes.

Pour plus d'informations, consultez la section Périodes de maintenance planifiées.

Périphérique

Le périphérique physique est un serveur 19 pouces avec un micrologiciel préinstallé préparé pour Checkmk qui peut être déployé immédiatement dans les centres de données. Il est livré avec une interface de configuration graphique qui élimine tout besoin de connaissances en Linux.

Pour plus d'informations, veuillez consulter la section Périphérique.

Plugin d'agent

Un plugin d'agent étend les fonctions de l'agent standard fourni avec Checkmk. Il s'agit d'un petit programme ou script appelé par l'agent Checkmk qui enrichit la sortie de l'agent avec des sections supplémentaires de données de supervision. Un exemple de plugin d'agent est la mise à jour de l'agent.

Pour en savoir plus, consultez la section Extension des agents avec des plugins.

Plugin de supervision

Les plugins de supervision sont des modules écrits en Python qui s'exécutent sur l'instance Checkmk et qui créent et évaluent les services d'un ordinateur hôte. Par exemple, le plugin de supervision df, disponible dans une instance à l'adresse ~/local/lib/python3/cmk_addons/plugins/<plug-in_family>/agent_based/ (lorsque vous utilisez l'API Check V2) ou ~/local/lib/check_mk/base/plugins/agent_based/ (lorsque vous utilisez l'API Check V1), crée des services pour les systèmes de fichiers montés existants d'un ordinateur hôte à partir des données d'un agent de l'instance, puis vérifie ces services par rapport à ces données, par exemple l'espace libre disponible.

En savoir plus sur les plugins de supervision.

Reconnaissance du service

Dès qu'un ordinateur hôte a été ajouté à la supervision, Checkmk détecte automatiquement tous les services disponibles pouvant être inclus dans la supervision et maintient cette liste à jour, même pendant le fonctionnement. Une reconnaissance du service peut également être lancée manuellement à tout moment via la configuration d'une fonction hôte.

Pour plus d'informations, consultez la section Comprendre et configurer les services.

Règle

Les règles constituent la base de la configuration des ordinateurs hôtes et des services dans Checkmk. Les règles d'un jeu de règles contrôlent toujours un aspect unique et précis d'un ordinateur hôte ou d'un service. Les règles peuvent être assorties de conditions et peuvent être « empilées » les unes sur les autres de manière arbitraire au sein d'un jeu de règles. L'évaluation des règles s'effectue ensuite de haut en bas, de sorte qu'il peut y avoir des règles standard pour les cas où aucune condition ne s'applique, ainsi que des règles très spéciales qui n'affectent qu'un hôte très spécifique. De nombreux jeux de règles dans Checkmk ont déjà des valeurs par défaut prédéfinies, de sorte que des règles supplémentaires ne doivent être créées que pour des exigences alternatives.

Pour en savoir plus, consultez la section Règles.

Service

Un service est un objet logique qui résume un ou plusieurs aspects d'un ordinateur hôte. Par exemple, la taille, l'utilisation et les tendances des systèmes de fichiers, l'utilisation du processeur, les températures, l'âge et le nombre de programmes en cours d'exécution, les ports, les capteurs, etc. À tout moment, chaque service de la supervision a l'un des statuts suivants : « OK », « WARN », « CRIT », « UNKNOWN » ou « PEND ». Il est toujours attribué à un seul ordinateur hôte et peut contenir une ou plusieurs métriques.

Pour plus d'informations, consultez la section Comprendre et configurer les services.

Snap-in

Les snap-ins, également appelés éléments de la barre latérale, sont les blocs individuels qui peuvent être placés dans la barre latérale, par exemple Overview et Master control. L'accès aux snap-ins est assuré par l'icône plus en bas de la barre latérale.

Pour plus d'informations, veuillez consulter la section Barre latérale.

SNMP

Le « Simple Network Management Protocol » est utilisé pour superviser et configurer des périphériques réseau tels que des routeurs, des commutateurs ou des pare-feu. Checkmk prend en charge ce protocole, mais comme il est relativement inefficace, vous ne devriez utiliser SNMP que sur des périphériques qui ne prennent pas en charge de meilleures alternatives de supervision, telles que les agents spéciaux.

Pour plus d'informations, veuillez consulter la section SNMP.

Supervision distribuée

Dans une supervision distribuée, l'ensemble du système de supervision se compose de plusieurs instances Checkmk interconnectées, c'est-à-dire d'une instance centrale et d'au moins une instance distante. Les données de toutes les instances sont affichées ensemble à un seul endroit, dans l'instance centrale. C'est ce qu'on appelle l'état central dans un système de supervision distribuée. La supervision distribuée peut être combinée en option avec une configuration centrale.

Plus d'informations sous Supervision distribuée.

Supervision distribuée avec configuration centrale

Avec une configuration centrale, la configuration de toutes les instances Checkmk connectées entre elles dans la supervision distribuée est effectuée à un seul endroit. La configuration est effectuée uniquement sur l'instance centrale, puis distribuée à toutes les instances distantes. Une configuration centrale comprend toujours un état central.

Pour plus d'informations, consultez la section Configuration centrale.

Tableau de bord

Un tableau de bord est un aperçu librement configurable composé de vues de la table et/ou de « dashlet ». Ces éléments sont disponibles, par exemple, sous forme de listes (telles que les problèmes d'ordinateurs hôtes), de visualisations de la série chronologique ou de petits « compteurs de vitesse » qui affichent des valeurs individuelles telles que la température du processeur.

Pour en savoir plus, consultez la section Tableaux de bord.

Utilisateur de l'automatisation

Compte spécial pour interroger et configurer Checkmk indépendamment de l'interface web, par exemple via API, ligne de commande, script ou service web. Par défaut, l'utilisateur de l'automatisation dispose d'un mot de passe d'automatisation choisi aléatoirement (mot de passe d'automatisation). Au moins un utilisateur de l'automatisation est déjà configuré dans une nouvelle instance Checkmk afin d'enregistrer l'agent auprès du serveur Checkmk pour le transfert de données cryptées TLS.

Plus d'informations sous Utilisateur de l'automatisation (pour les services web).

Vue de la table

Outre les tableaux de bord, les vues sont les affichages les plus fréquemment utilisés des ordinateurs hôtes, des services et d'autres composants dans l'interface Checkmk. Ces vues sont affichées sous forme de tableaux avec des attributs pertinents pour le contexte actuel. Par exemple, « All hosts » et « Host problems » sont des vues dans la supervision. Les vues standard fournies peuvent être personnalisées dans leur affichage et peuvent également servir de base à de nouvelles vues. Il est également possible de créer des vues à partir de zéro.

Pour plus d'informations, consultez la section Vues des ordinateurs hôtes et des services.

WATO

Setup L'outil d'administration web (WATO) était l'interface graphique utilisée pour configurer Checkmk jusqu'à la version 1.6.0. Avec l'introduction de WATO, les utilisateurs ont pu pour la première fois personnaliser Checkmk via une interface web au lieu d'utiliser des fichiers de configuration. WATO a été remplacé dans la version 2.0.0 par le menu « Interface utilisateur web » dans la barre de navigation.

Pour plus d'informations, veuillez consulter le menu Configuration.

Werk

Le développement du logiciel Checkmk est organisé en « Werks ». Chaque modification, correction de bug ou nouvelle fonctionnalité ayant un impact sur l'expérience utilisateur est enregistrée dans un Werk distinct, accompagnée de notes sur les impacts et les incompatibilités éventuelles. La liste des Werks est disponible directement dans Checkmk via le menu « Help » dans la barre de navigation et sur la page d'accueil de Checkmk.

Plus d'informations sous Werks.
